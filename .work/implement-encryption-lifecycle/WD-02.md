## Summary

Ciphertext envelope has two layers:

1. **Per-field envelope** — algorithm-variant byte layout with a 4-byte
   plaintext DEK version tag and per-variant MAC placement. Canonically
   specified by `encryption.ciphertext-envelope` (APPROVED). Implementation
   exists; annotation and round-trip test coverage need closure.
2. **SSTable footer scope signalling** — the `(tenantId, domainId, tableId)`
   scope metadata + DEK version set recorded in each SSTable's footer so
   the reader can (a) locate the correct DEK without cross-scope leakage,
   and (b) reject cross-scope reads before decryption. This is the
   "signalling mechanism for the reader to locate which DEK was used"
   half of the WD.

Layer 2 requires new sstable-side spec work, architect decisions on the
footer format, and a catalog API for Table-handle → scope derivation.

## Acceptance Criteria

**Per-field envelope layer**
- `encryption.ciphertext-envelope` R1–R4 direct-annotated against
  implementation (all four variants: Opaque, Deterministic, OrderPreserving,
  DistancePreserving); test gaps closed with `@spec` annotations
- `encryption.primitives-lifecycle` R22, R22a, R23 direct-annotated
  (pointer requirements to ciphertext-envelope)
- Round-trip test: encrypt → persist → decrypt via the envelope across
  every variant, asserting byte-identity across MemTable/WAL/SSTable tiers

**Footer scope signalling layer**
- `sstable.footer-encryption-scope` spec authored and APPROVED (new spec),
  defining footer-layout extension, backward-compat semantics, and reader
  scope-validation protocol
- Architect decisions landed on (a) footer format evolution (v3 → v4
  bump vs. optional-extension in v3), (b) catalog API for Table-handle
  scope exposure, (c) tamper-resistance posture for footer scope bytes
- `encryption.primitives-lifecycle` R22b, R23a, R24 direct-annotated
  once footer scope implementation lands (WD-02 implementation stage)
- Cross-scope read adversarial test: mis-routed SSTable is rejected with
  `IllegalStateException` before any DEK lookup occurs
- Table catalog exposes a scope accessor consumed by the encryption
  read path

## Implementation Notes

- Footer format change may necessitate an SSTable format version bump
  (current v3 per `sstable.v3-format-upgrade`). Evaluate optional-extension
  vs. v4 bump via ADR — backward-compat is required (pre-encryption
  SSTables must remain readable).
- Scope bytes in the footer must survive tamper attempts that would
  otherwise succeed in re-pointing a DEK lookup across scopes. Decide
  whether to sign/CRC the scope block or rely on R22b's pre-decrypt
  comparison alone. ADR territory.
- Catalog scope exposure: likely `Table.scope()` returning a new
  `TableScope(tenantId, domainId, tableId)` record. Must not leak
  privileged data; exposed only to encryption-layer internals.
