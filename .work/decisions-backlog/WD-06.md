## Summary

Database engine surface — handle lifecycle, cross-table operations, and catalog
management are the engine's public contract. Handle items (timeout, priority,
budgets) can proceed independently, but remote serialization, catalog replication,
and table migration depend on networking (Cluster 5).

Decisions in this cluster: handle-timeout-ttl, handle-priority-levels,
cross-table-handle-budgets, message-serialization-format, cross-table-transactions,
remote-serialization-protocol, atomic-multi-table-ddl, catalog-replication

## Acceptance Criteria

- Each decision slug in this cluster has an accepted ADR or is explicitly
  closed/re-deferred
- Implementation notes recorded for any that need follow-up features

## Implementation Notes

**Effort classification:**
- Gap-fills: handle-timeout-ttl
- Minor features: handle-priority-levels, cross-table-handle-budgets,
  message-serialization-format
- Full features: cross-table-transactions, remote-serialization-protocol,
  atomic-multi-table-ddl, catalog-replication

**Approach:** Handle gap-fill and minor features can start immediately (no
networking dependency). message-serialization-format is a dependency for
remote-serialization-protocol and catalog-replication — resolve it early.
Full features require networking stability and full /architect pipeline.
Note: table-migration-protocol is in Cluster 7 (depends on catalog-replication
+ partition-replication-protocol).
