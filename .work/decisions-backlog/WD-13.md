## Summary

Catalog replication across cluster nodes, the table migration protocol built
on top, and scan lifecycle management (snapshot binding and GC watermark
leasing). catalog-replication was deferred pending F37 (Catalog Operations).
table-migration-protocol was deferred pending both catalog-replication and
partition-replication-protocol (F32). scan-snapshot-binding and
scan-lease-gc-watermark were deferred pending scatter-gather proxy
specification — now covered by F21 and F39.

Decisions in this cluster: catalog-replication, table-migration-protocol,
scan-snapshot-binding, scan-lease-gc-watermark

## Acceptance Criteria

  closed/re-deferred

## Implementation Notes

**Effort classification:**
- Minor features: scan-snapshot-binding
- Full features: catalog-replication, table-migration-protocol,
  scan-lease-gc-watermark

**Approach:** scan-snapshot-binding first — it's a focused decision on
point-in-time consistency vs best-effort latest-snapshot semantics for
distributed scans, directly informing F39's snapshot degradation behavior.
scan-lease-gc-watermark depends on scan-snapshot-binding (chain). Then
catalog-replication — how the table catalog (F37) is replicated to cluster
nodes. table-migration-protocol depends on catalog-replication being
resolved.

**Internal dependency chain:** scan-snapshot-binding → scan-lease-gc-watermark;
catalog-replication → table-migration-protocol.

**Key specs to reference:** F37 (Catalog Operations), F39 (Distributed
Pagination), F32 (Partition Replication), F21 (Scatter-Gather Flow Control)
