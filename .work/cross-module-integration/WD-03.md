## Summary

Wire Table.query() through StringKeyedTable to QueryExecutor so that
in-process engine users can execute structured queries. Currently
Table.query() throws UnsupportedOperationException because TableQuery has
a private constructor and cannot be instantiated outside jlsm.table. This
obligation is blocked on F10 query infrastructure — WD-01 (fulltext
integration) validates the index wiring pattern that query binding depends on.

## Acceptance Criteria

- F05.R37 satisfied: Table.query() returns a functional TableQuery that
  executes against the table's schema and registered indices
- TableQuery is bindable (package-private constructor or factory accessible
  to StringKeyedTable)
- StringKeyedTable carries schema + IndexRegistry references needed for
  query binding
- QueryExecutor.execute() produces correct results through the query path
- Obligation OBL-F05-R37 resolved in _obligations.json

## Implementation Notes

**Four-part resolution (from obligation description):**
1. Add `TableQuery<String> query()` to JlsmTable.StringKeyed (cross-module
   API change in jlsm-table)
2. Refactor StringKeyedTable to carry schema + IndexRegistry
3. Make TableQuery bindable — package-private constructor or static factory
4. Wire execute() to QueryExecutor with index-backed predicate resolution

**Dependency on WD-01:**
- WD-01 validates that index implementations are wired through IndexRegistry.
  Query binding needs the same wiring to resolve index-backed predicates.
- The pattern established in WD-01 (factory injection into IndexRegistry)
  informs how QueryExecutor discovers available indices.

**Tests needed:**
- Simple query: equality predicate on indexed field
- Multi-predicate query: AND/OR combinations
- Unindexed field: falls back to scan
- Schema mismatch: query against non-existent field rejected
- Empty result set handling
