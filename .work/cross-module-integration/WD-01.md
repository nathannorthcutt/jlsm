## Summary

Wire LsmFullTextIndex from jlsm-indexing through the jlsm-core interface
boundary so that FullTextFieldIndex delegates to a real implementation
instead of throwing UnsupportedOperationException. Currently IndexRegistry
validates FULL_TEXT index definitions correctly, but any table with a
FULL_TEXT index fails on first write.

## Acceptance Criteria

- query.full-text-index.R1 satisfied: onInsert propagates document fields to LsmFullTextIndex
- query.full-text-index.R2 satisfied: onUpdate propagates field changes to LsmFullTextIndex
- query.full-text-index.R3 satisfied: onDelete removes document from LsmFullTextIndex
- query.full-text-index.R4 satisfied: lookup returns results via LsmFullTextIndex
- query.full-text-index.R5 satisfied: supports() returns true for FULL_TEXT predicates
- query.full-text-index.R6 satisfied: close() cleans up LsmFullTextIndex resources
- query.index-types.R5 (partial): tables with FULL_TEXT indices accept writes without
  UnsupportedOperationException
- Obligation OBL-query.index-types-fulltext resolved in _obligations.json

## Implementation Notes

**Module boundary crossing:**
- jlsm-indexing exports LsmFullTextIndex with its own on-disk format
- jlsm-core defines FieldIndex interface (FullTextFieldIndex is the stub)
- FullTextFieldIndex must delegate to LsmFullTextIndex via constructor
  injection or factory pattern
- IndexRegistry needs to resolve the concrete implementation at table
  creation time

**Key files (from obligation description):**
- FullTextFieldIndex.java in jlsm-core — the stub to replace
- IndexRegistry.java — already validates field type constraints
- LsmFullTextIndex in jlsm-indexing — the real implementation

**Tests needed:**
- End-to-end: create table with FULL_TEXT index, insert documents, query
- Mutation lifecycle: insert → update → delete → verify index state
- Field type validation: FULL_TEXT on non-string field rejected
- Close lifecycle: index resources released on table shutdown
