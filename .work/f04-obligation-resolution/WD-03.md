## Summary

Fix the remote request payload format (R68) and implement parallel partition
scatter for scans (R77). RemotePartitionClient payloads currently lack
table name and partition ID, so remote nodes cannot route requests. Scans
issue partition requests sequentially instead of in parallel.

## Acceptance Criteria

- F04.R68 satisfied: RemotePartitionClient payloads include table name
  and partition identifier; remote node can route QUERY_REQUEST to the
  correct table
- F04.R77 satisfied: ClusteredTable.scan issues partition requests in
  parallel via CompletableFuture-based fanout; sequential blocking per
  partition is eliminated
- Obligations OBL-F04-R68-payload-table-id and OBL-F04-R77-parallel-scatter
  resolved in _obligations.json

## Implementation Notes

**R68 — Payload format:**
- Add table-name and partition-id fields to the request payload format
- Build the remote-side QUERY_REQUEST dispatcher that routes incoming
  requests to the correct local table
- Wire dispatcher into the message handling loop
- Tests: multi-table cluster where requests target different tables

**R77 — Parallel scatter:**
- Refactor getRange to return CompletableFuture<Iterator<TableEntry<String>>>
  instead of eagerly materializing List<TableEntry<String>>
- ClusteredTable.scan: fan out partition requests concurrently, collect
  via CompletableFuture.allOf() or barrier
- Merge results maintaining sort order across partitions
- Tests: latency improvement under multi-partition scans, correct ordering

**Spec reference:** F04 requirements R68, R77
