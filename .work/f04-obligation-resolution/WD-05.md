## Summary

Implement split-brain handling (R41-43), grace-gated rebalancing (R47-50),
and partition pruning for scans (R63). These are the fault tolerance and
operational intelligence features that depend on consensus (WD-04) being
in place. Without split-brain handling the engine continues accepting
writes during network partitions. Without grace-gated rebalancing, every
transient node departure triggers full partition redistribution.

## Acceptance Criteria

- F04.R41 satisfied: engine transitions to read-only mode when quorum
  is lost; write operations return an appropriate error indicating
  degraded state
- F04.R42 satisfied: during quorum loss, the engine proactively retries
  contact with seed nodes to detect partition heal
- F04.R43 satisfied: on partition heal, conflicting views are merged
  using the reconciliation rules (higher incarnation wins; equal
  incarnation resolves by severity DEAD > SUSPECTED > ALIVE)
- F04.R47 satisfied: when a member departs, its partitions are held
  unowned for the configured grace period before redistribution
- F04.R48 satisfied: on grace period expiration, only the departed
  member's partitions are redistributed; partitions among still-live
  members are not moved (differential HRW or stable-ownership-with-deltas)
- F04.R50 satisfied: if a departed member rejoins within the grace
  period, the pending rebalance is cancelled and the member reclaims
  its previous partition assignments
- F04.R63 satisfied: ClusteredTable.scan uses partition-key predicates
  to narrow fanout to intersecting owners instead of scattering to
  every live member
- Obligations OBL-F04-R41-43-split-brain, OBL-F04-R47-50-grace-gated-rebalance,
  and OBL-F04-R63-partition-pruning resolved in _obligations.json

## Implementation Notes

**R41-43 — Split-brain:**
- Add quorum monitoring to ClusteredEngine (check on every view change)
- Implement read-only mode: reject mutations with QuorumLostException,
  allow reads
- Seed retry loop: periodic background task during quorum loss
- View merge: on reconnection, merge views by incarnation comparison
- Tests: network partition simulation, write rejection during split,
  recovery after heal

**R47-50 — Grace-gated rebalancing:**
- GracePeriodManager already tracks departures/returns but doesn't gate
  rebalancing. Wire it into ClusteredEngine.onViewChanged
- On departure: schedule deferred rebalance instead of immediate cache
  invalidation
- On grace expiry: compute differential ownership (only departed member's
  partitions move)
- On rejoin within grace: cancel pending rebalance, restore assignments
- Tests: transient departure within grace, permanent departure past grace,
  differential redistribution correctness

**R63 — Partition pruning:**
- Add partition-key boundary tracking to RendezvousOwnership
- ClusteredTable.scan: extract key range from query predicate, ask
  RendezvousOwnership which owners intersect the range
- Fan out only to intersecting owners
- Tests: range scan touching subset of partitions, full-table scan
  still fans out to all
