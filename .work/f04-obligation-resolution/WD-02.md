## Summary

Fix the ClusteredEngine builder and join sequence (R56-57-79) and add
local short-circuit routing in ClusteredTable (R60). The builder currently
does not accept a DiscoveryProvider, forcing callers to start the membership
protocol externally. ClusteredTable always routes through RemotePartitionClient
even for locally-owned partitions.

## Acceptance Criteria

- F04.R56 satisfied: ClusteredEngine.Builder accepts a mandatory
  DiscoveryProvider parameter
- F04.R57 satisfied: ClusteredEngine orchestrates the join sequence
  (discovery.register → membership.start) with rollback on partial failure
- F04.R79 satisfied: engine join sequence is atomic — partial failure
  cleans up registered discovery and stops membership
- F04.R60 satisfied: ClusteredTable checks if resolved owner equals
  localAddress and routes directly to the local engine table, bypassing
  RemotePartitionClient
- Obligations OBL-F04-R56-57-79-engine-join and OBL-F04-R60-local-short-circuit
  resolved in _obligations.json

## Implementation Notes

**R56-57-79 — Engine join:**
- Expand ClusteredEngine.Builder to require DiscoveryProvider
- Add join() method: discovery.register(localAddress) → membership.start()
- Add rollback: if membership.start() fails, discovery.deregister()
- Update all test fixtures that construct ClusteredEngine

**R60 — Local short-circuit:**
- ClusteredTable constructor gains a reference to the local Engine
- Route decision: if owner.equals(localAddress), use localEngine.getTable(name)
  directly instead of RemotePartitionClient
- Propagate through ClusteredEngine.createTable()
- Tests: verify local operations never touch transport layer
